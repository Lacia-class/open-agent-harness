# Meta-Recursive Design

Use this protocol for designing systems, SOPs, skills, review methods, agent workflows, or other repeatable methods.

## Boundary

Meta-recursion is not default ceremony. Use it only when:

- the design object is itself a method, process, standard, workflow, or harness
- the design process and design object share structure
- applying the method to itself can produce an executable change
- there is a clear stopping condition

Do not use it when reality validation is more urgent than method reflection.

## Principles

### 1. Method Applies To Itself

Turn the method's own diagnostic axes back onto the method. If that produces no new constraint, patch, failure case, or validation path, stop recursing.

### 2. Process Is Prototype

The design conversation is the first run of the system. Treat user reactions, confusion, corrections, and fatigue signals as validation data, while avoiding psychological overreach.

### 3. Preserve Tension

When two needs conflict, avoid premature tradeoff. Prefer:

- sequencing
- reversible steps
- explicit risk gates
- adapter boundaries
- partial delivery with known unmeasured areas

### 4. Cross-Domain Mapping

When stuck, abstract the structure and look for an analogous solution in another domain. Keep only the relation structure, not the domain constants.

## Output Requirement

Every use must produce at least one executable change:

- design decision
- boundary
- checklist
- patch
- validation plan
- failure mode
- functional incompleteness statement
