# Compound Interest Engineering

Use this protocol after a workflow, experiment, or long collaboration reveals something that might compound across future tasks.

## Goal

Extract reusable assets without turning one successful anecdote into an overfit rule.

## Evidence Layers

| Evidence | Examples | Limitation |
|---|---|---|
| Interaction | user corrections, pacing, fatigue, trust shifts | often N=1 |
| Process | tool order, review rounds, issue count convergence | environment-dependent |
| Artifact | diffs, files, tests, screenshots, probes | proves artifact, not universal method |
| External reality | live service response, browser state, real API output | high cost and risk |

Evidence must match the asset being created.

## Extraction Steps

1. List sample constants that must not become rules.
2. Identify transferable variables.
3. Explain the mechanism, not just the observed correlation.
4. Draw at least one compounding loop: A improves B, B improves C, C improves A.
5. Package as one of:
   - skill / protocol
   - template
   - adapter
   - checklist
   - private memory
   - candidate note

## Mechanism Classifiers

When a phrase, analogy, or classifier improves attribution quality, first store it as a diagnostic axis.

Keep two layers:

- **Mechanism layer**: what causal escape, acceptance pollution, cognitive error, or workflow distortion it detects.
- **Example layer**: concrete checklist items and cases used only for boundary calibration.

Only promote a diagnostic axis to a default strategy after it improves judgment across isolated, out-of-distribution, or repeated tasks.
